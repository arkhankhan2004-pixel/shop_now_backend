Capstone Project 2026

ShopNow

Industrial MySQL Solution

An advanced e-commerce ecosystem integrating high-performance Flutter interfaces with a scalable Node.js backend and a strictly normalized MySQL data core.

Frontend Architecture

Flutter Reactive Framework

The client-side experience of ShopNow is built on the robust Flutter framework. We utilized a widget-centric approach to build a highly interactive and intuitive user interface. By implementing centralized state management, we ensure that every interaction—from category selection to checkout—is handled with zero latency. The UI follows Material 3 guidelines to provide a modern, premium feel across all mobile platforms.

Adaptive UX

Responsive Layout Control

Ensuring cross-device compatibility was a primary objective. We engineered a responsive layout engine that utilizes MediaQuery and custom grid delegates. This allows the application to dynamically transition from a single-column layout on small phones to a complex five-column grid on high-resolution laptop screens. This flexibility ensures that the storefront remains visually balanced and highly usable on any hardware.

Server Infrastructure

Node.js & Express Performance

Our server-side logic is powered by Node.js, an event-driven, non-blocking I/O runtime that is perfect for high-traffic e-commerce environments. We used Express.js to build a modular API structure that handles authentication, product management, and complex transaction logic. This backend ensures that even during high concurrent load, the system maintains sub-millisecond response times for all client-side requests.

API Orchestration

RESTful Communication Flow

The bridge between the client and the data is a series of strictly typed RESTful endpoints. We designed this layer to handle JSON payloads efficiently, ensuring a decoupled architecture where the frontend and backend can evolve independently. This standardized communication protocol allows us to implement features like real-time inventory updates and secure order processing with maximum architectural clarity.

Database Schema: Table 1

User Management (users)

CREATE TABLE users (
  id INT AUTO_INCREMENT PRIMARY KEY,
  firebase_uid VARCHAR(255) UNIQUE NOT NULL,
  name VARCHAR(100) NOT NULL,
  email VARCHAR(100) UNIQUE NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

Technical Working

Identity Synchronization

The `users` table is the source of truth for customer identity. We store a unique `firebase_uid` to map authenticated mobile users to their relational database records. This allows the system to maintain a persistent profile, including shipping details and purchase history, without storing sensitive password data locally. It ensures a seamless bridge between modern OAuth providers and our industrial MySQL backend.

Database Schema: Table 2

Catalog Engine (products)

CREATE TABLE products (
  id INT AUTO_INCREMENT PRIMARY KEY,
  name VARCHAR(255) NOT NULL,
  price DECIMAL(10, 2) NOT NULL,
  category VARCHAR(100),
  discount_percent INT DEFAULT 0,
  image_url VARCHAR(500)
);

Technical Working

Inventory & Sales Logic

The `products` table manages a high-volume catalog of 100+ unique items. By using the `DECIMAL` type, we guarantee financial precision across all transactions. The `discount_percent` field is utilized by our backend logic to calculate dynamic sale prices on-the-fly. This allows the admin to run store-wide promotions that reflect instantly on the user's home screen, powered by efficient SQL indexing.

Database Schema: Table 3

Transaction Hub (orders)

CREATE TABLE orders (
  id INT AUTO_INCREMENT PRIMARY KEY,
  user_id INT,
  total_amount DECIMAL(10, 2) NOT NULL,
  status VARCHAR(50) DEFAULT 'Pending',
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

Technical Working

Transactional Lifecycle

The `orders` table tracks every sale from inception to delivery. By using a Foreign Key linked to the `users` table with `ON DELETE CASCADE`, we ensure that data remains clean and orphaned records are never created. This table manages the `status` of each purchase, allowing for real-time tracking of order fulfillment states, which is a critical feature for any production-grade e-commerce application.

Database Schema: Table 4

Relational Link (order_items)

CREATE TABLE order_items (
  id INT AUTO_INCREMENT PRIMARY KEY,
  order_id INT,
  product_id INT,
  quantity INT NOT NULL,
  price DECIMAL(10, 2) NOT NULL,
  FOREIGN KEY (order_id) REFERENCES orders(id) ON DELETE CASCADE
);

Technical Working

Normalization & Scaling

The `order_items` table is the relational bridge that allows a single order to contain an unlimited variety of products. By storing the `price` at the moment of sale, we protect historical records from future price fluctuations in the master catalog. This Many-to-Many resolution is essential for maintaining a strictly normalized database that can handle complex queries without performance degradation.

Store Management

The Admin Control Suite

Efficiency was a primary goal for the administrator's experience. Our dashboard provides real-time access to the entire MySQL catalog. Admins can perform bulk operations, such as multi-deleting obsolete inventory or updating stock status across multiple hubs. The interface is optimized with long-press multi-selection and atomic update calls, ensuring that the store remains fresh and responsive for all users.

SQL Insights

Real-time Inventory Visuals

Strategic store decisions require live data. The Admin panel includes a category-wise analytics hub that executes summary SQL queries instantly. Administrators can visualize exactly how many products are live in each category (Tech, Beauty, Groceries, etc.) with a single glance. This real-time feedback loop between the MySQL data and the Flutter UI is what makes ShopNow a truly professional solution.
